## 使い勝手の評価

### 良い点

1. **出力の見やすさ**: Rust 系ツール風の `-->` 付きコードスニペット表示で、問題箇所が一目でわかる。ファイル名・行番号・列番号が正確。
2. **サマリーテーブル**: 実行末尾のファイル別 errors/warnings テーブルが全体感の把握に非常に有効。
3. **`= help:` メッセージ**: 一部ルールで具体的な修正方針が提示される (`run-inputs-context-direct-use` など)。
4. **自動修正の品質**: `--fix` による修正が正確。`${{ env.* }}` → `${VAR}` の変換は env ブロックの存在を考慮している。コンテナイメージの SHA256 ピンも正しく解決される。
5. **dry-run サポート**: `--fix --dry-run` で unified diff が出力され、修正前に確認できる。
6. **コンフィグの柔軟性**: ファイル × ルール単位で除外でき、グローバル無効化せずに済む。
7. **検出精度**: 61 ルール中 56 がデフォルト有効で、過不足のないバランス。
8. **パフォーマンス**: 123 ファイルを 1 秒未満で解析 (verbose 表示で各ファイル 0.1-8ms)。
9. **`seiton rules`**: 利用可能な全ルールとその状態を一覧表示でき、コンフィグ調整の参考になる。
10. **`seiton validate-config`**: コンフィグの妥当性を事前検証できる。

### 改善の余地

#### 1. `bot-conditions` ルールの精度

`github.actor != 'dependabot[bot]'` は非常に一般的なパターンだが、常に警告される。Dependabot の actor チェックは GitHub 公式ドキュメントでも推奨されており、false positive 感が強い。スプーフ可能性の警告は理解できるが、severity を info にするか、dependabot 特有パターンを除外オプションとして提供すると良い。

**具体的な出力例:**

```
warning[bot-conditions]: bot actor check uses spoofable context; prefer github.event.pull_request.user.login, github.event.pull_request.user.id, or another trigger-author context
  --> .github/workflows/auto-doc.yaml:11:9
     |
   11 |     if: ${{ github.actor != 'dependabot[bot]' }}
     |         ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
     |
```

**再現ワークフロー** (`.github/workflows/auto-doc.yaml`):

```yaml
name: auto doc
on:
  pull_request:
    branches: ["main"]
  push:
    branches: ["main"]

jobs:
  embed-code:
    if: ${{ github.actor != 'dependabot[bot]' }}
    runs-on: ubuntu-24.04
    steps:
      - uses: actions/checkout@v4
```

**問題点**: `github.actor` を使った dependabot 除外は GitHub Docs でも紹介されている標準的なパターン。代替の `github.event.pull_request.user.login` は `push` イベントでは使えないため、`github.actor` が唯一の選択肢になるケースがある。ルールが一律に警告するのは過剰。

#### 2. `run-inputs-context-direct-use` の自動修正

複雑な式 (`inputs.tag || (...)`) を含む場合の自動修正が未対応。help メッセージに「env ブロックへの移行」が提案されるが、自動修正で対応できると更に便利。

**具体的な出力例:**

```
error[run-inputs-context-direct-use]: run script must not reference ${{ inputs.* }} or ${{ github.event.inputs.* }} directly; map inputs to env and use shell variables instead (e.g. ${NAME}, $NAME, or $env:NAME)
  --> .github/workflows/create-release.yaml:36:26
     |
   36 |         run: echo "value=${{ inputs.tag || (github.event_name == 'pull_request' && '0.1.0-test' || github.ref_name) }}" | tee -a "$GITHUB_OUTPUT"
     |                          ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
     |
   = help: consider moving the entire expression to an env: block and referencing the shell variable instead
```

**再現ワークフロー** (`.github/workflows/create-release.yaml`):

```yaml
jobs:
  create-release:
    runs-on: ubuntu-24.04
    timeout-minutes: 10
    steps:
      - name: Setup tag
        id: tag
        run: echo "value=${{ inputs.tag || (github.event_name == 'pull_request' && '0.1.0-test' || github.ref_name) }}" | tee -a "$GITHUB_OUTPUT"
```

**期待する自動修正結果:**

```yaml
    steps:
      - name: Setup tag
        id: tag
        run: echo "value=${TAG_VALUE}" | tee -a "$GITHUB_OUTPUT"
        env:
          TAG_VALUE: ${{ inputs.tag || (github.event_name == 'pull_request' && '0.1.0-test' || github.ref_name) }}
```

**問題点**: `= help:` メッセージで修正方針を提示するなら、同じロジックで自動修正を実装できるはず。式を env ブロックに移し、変数名をステップ output 名や式のルートから推定すれば機械的に修正可能。

### ログからの状況把握

| 項目 | 評価 |
|------|------|
| エラー/警告の区別 | ◎ `error[rule-id]` / `warning[rule-id]` で明確 |
| 問題箇所の特定 | ◎ ファイル:行:列 + コードスニペット |
| 修正方針の提示 | ○ 一部ルールのみ `= help:` あり。全ルールにあると理想的 |
| 全体感の把握 | ◎ 末尾のサマリーテーブルで一覧性高い |
| 修正結果の確認 | ◎ `Fixed X of Y issues in Z files (N remaining)` が明確 |
| 除外の反映確認 | ○ verbose で `N suppressed` と表示されるが、通常出力では見えない |

## 結論

seiton は GitHub Actions ワークフローの lint ツールとして完成度が高い。検出精度・自動修正品質・出力の見やすさのバランスが良く、素直に使える。特に `--fix --dry-run` → `--fix` のワークフローが直感的。コンフィグのファイル × ルール単位の除外も柔軟で、ラボリポジトリのようにデモ用アンチパターンを含む環境でも適切に運用できる。

主要な改善点は `bot-conditions` の false positive 率と、Agentic Workflow の自動除外サポート。
